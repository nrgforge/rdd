# Reflections: 004 — Orientation Document as Artifact Entry Point

## "A Comprehensible Marriage"

The user's phrase — "a comprehensible marriage of the system-design and product-discovery" — captures the document's identity more precisely than the essay's Diataxis framing. The orientation document is not a third perspective alongside product and technical. It is the bridge between the two perspectives that already exist. This distinction matters: a third perspective would need its own content and voice; a bridge derives its content from the two sides it connects.

## Real-World Feedback as Validation

The motivation came from circulating RDD artifacts with both technical and product stakeholders. The gap was experienced, not theorized — people lacked a single coherent high-level orientation. This is the strongest possible validation for the document's existence: the absence was felt before the solution was designed.

## "A Doc Worth Reading"

The user's emphasis on the document being "worth reading" connects to the synthesis phase's "worth the calories" principle. The orientation document shares this constraint. If it reads like boilerplate — generic project overview language that could apply to any system — it fails. The document must be specific enough to this system that reading it genuinely orients the reader. This is a quality bar, not a length bar.

## README Disambiguation

The README intuition turned out to be about purpose, not just naming. README serves installation, usage, and contribution — operational concerns. The orientation document serves comprehension — "what is this system and why does it exist." These are fundamentally different questions. A README for the docs/ folder could coexist with the orientation document, serving its conventional purpose (what's in this folder, how to navigate it) while the orientation document serves its explanatory purpose (what the system is, who it's for, what's decided).

## Open: Naming Still Unresolved

The naming question remains genuinely open. Candidates discussed (BRIEFING, PRIMER, BRIEF, DIGEST) each capture part of the document's identity but none captures the "living marriage" quality the user described. The right name may need to emerge from use. Worth noting: the document's *function* is clear even without a settled name — proceed to domain modeling and let naming resolve through practice.
