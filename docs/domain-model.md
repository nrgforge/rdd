# Domain Model: Pedagogical RDD

## Concepts (Nouns)

| Term | Definition | Related Terms |
|------|-----------|---------------|
| Opacity Problem | The compounding gap between what AI generates and what the user understands, where each layer of uncomprehended system becomes a foundation for the next. | Metacognitive Illusion, Tacit Knowledge |
| Epistemic Action | An action performed to build understanding of a problem, not to advance toward a solution. Term from Kirsh & Maglio (1994). Contrasts with Pragmatic Action. | Pragmatic Action, Epistemic Gate |
| Pragmatic Action | An action performed to advance toward a goal or produce output. Can be safely automated without learning loss. | Epistemic Action |
| Epistemic Act | A specific, structured cognitive activity performed by the user at an RDD gate, drawn from one of six pedagogical frameworks. The concrete realization of epistemic action preservation within the methodology. | Epistemic Gate, Self-Explanation, Elaborative Interrogation, Retrieval Practice, Articulation, Reflection |
| Gate | A transition point between RDD phases where the user engages with the phase's artifact before proceeding. | Approval Gate, Epistemic Gate, Phase |
| Approval Gate | A gate where the user reviews and accepts an artifact without producing anything. The current RDD pattern. Does not activate learning mechanisms. Avoid: this is the pattern being replaced. | Gate |
| Epistemic Gate | A gate that requires the user to perform one or more Epistemic Acts — producing a targeted reconstruction, prediction, or explanation — before proceeding. The proposed replacement for Approval Gates. User-facing name: Reflection Time. | Gate, Epistemic Act, Grounding Move, Reflection Time |
| Artifact | A document produced by an RDD phase (essay, product discovery document, domain model, ADR, system design, roadmap, field guide, code). | Epistemic Artifact, Phase |
| Epistemic Artifact | An artifact that serves simultaneously as a project deliverable and as an instrument of knowledge advancement. Term from Scardamalia & Bereiter. All RDD artifacts should be treated as epistemic artifacts. | Artifact |
| Phase | A discrete stage in the RDD pipeline (RESEARCH, PRODUCT DISCOVERY, MODEL, DECIDE, ARCHITECT, BUILD, optionally PLAY, and optionally SYNTHESIS). Each phase produces an artifact and ends at a gate — except SYNTHESIS and PLAY, which are phases where the activity itself is the epistemic act (ADR-016 principle). | Gate, Artifact, Pipeline, Synthesis, Play |
| Pipeline | The ordered sequence of RDD phases from RESEARCH through BUILD, with optional PLAY and SYNTHESIS phases after BUILD. Epistemic acts at each gate compound across the pipeline. The synthesis essay closes the epistemic loop opened by RESEARCH. Play and discover form epistemic bookends around the pragmatic center. Can be scoped to a subsystem or feature rather than a whole project. | Phase, Synthesis, Play, Scoped Cycle, Deep Work Tool |
| Self-Explanation | Epistemic act: the user explains artifact elements in their own words. Constructive, integrative, and error-correcting. From Chi (1994). Strongest general-purpose epistemic act. | Epistemic Act |
| Elaborative Interrogation | Epistemic act: the user answers "why does this make sense?" Lightest-weight intervention. Forces connection of new material to prior knowledge. From Pressley (1987). | Epistemic Act |
| Retrieval Practice | Epistemic act: the user reconstructs key elements from memory before re-reading the artifact. Strongest for long-term retention. From Roediger & Karpicke (2006). | Epistemic Act |
| Articulation | Epistemic act: the user explicitly states their knowledge and reasoning about the artifact. From Collins, Brown, & Newman (1987). Transforms approval into active knowledge construction. | Epistemic Act |
| Reflection | Epistemic act in two modes. Reflection-on-action: examining what was learned after a phase completes. Reflection-in-action: thinking about the work while doing it (e.g., predicting test outcomes during BUILD). From Schon (1983). | Epistemic Act |
| Fading | The gradual withdrawal of AI-generated scaffolding as the user's expertise grows. Across RDD cycles, the generative burden shifts from AI toward user. | Scaffolding, Dreyfus Stage |
| Scaffolding | Temporary support that enables the user to engage with tasks beyond their current ability. RDD's phased pipeline provides structural scaffolding. Must fade to avoid becoming a permanent crutch. Graduation applies this principle at the methodology level — RDD's artifact structure is itself scaffolding that can be removed once knowledge is integrated. | Fading, Graduation |
| Metacognitive Prompt | An explicit question embedded in a gate that triggers the user's reflection on their own understanding ("What changed? What surprised you? What would you explain differently?"). The mechanism that determines whether gates produce learning or just artifacts. | Epistemic Gate, Reflection |
| Metacognitive Illusion | The false sense that understanding is strong because AI output is strong. Output quality and human understanding are independent variables. Term derived from Bjork. | Opacity Problem, Approval Gate |
| Grounding | The collaborative process of establishing shared understanding between user and AI. Requires active verification, not just approval. From Clark & Brennan (1991). | Grounding Move, Common Ground |
| Grounding Move | A specific communicative act that verifies shared understanding. Epistemic acts at gates function as grounding moves — they signal to the AI what the user actually understood, not just that the output looked right. | Grounding, Epistemic Act |
| Common Ground | The mutual knowledge, beliefs, and assumptions shared between user and AI at any point in the pipeline. Enriched by epistemic acts; eroded by approval-only gates. | Grounding |
| Tacit Knowledge | Knowledge the user holds but has not articulated. A primary source of requirements defects. Epistemic acts force tacit knowledge into explicit form, preventing hidden defects. | Opacity Problem, Grounding |
| Desirable Difficulty | A condition that makes learning feel harder but produces better long-term retention. Epistemic acts introduce targeted desirable difficulties at gates. From Bjork (1994). | Epistemic Act |
| Cognitive Level | The depth of cognitive engagement, mapped to Bloom's taxonomy (Remember, Understand, Apply, Analyze, Evaluate, Create). Approval gates engage only Evaluate. Epistemic gates engage Analyze and Create. | Epistemic Gate, Approval Gate |
| Dreyfus Stage | The user's current stage of skill acquisition (Novice → Expert). Determines how much scaffolding is appropriate and when fading should occur. | Fading, Scaffolding |
| Maintenance Cliff | The point where initial velocity advantages of AI-assisted development invert due to accumulated opacity. Debugging and extending uncomprehended code takes longer than manual development would have. The concrete, measurable consequence of the Opacity Problem. | Opacity Problem, Approval Gate |
| Context Window Ceiling | The hard constraint where system complexity exceeds what AI can reason about holistically. At this point, the human's structural understanding becomes the irreplaceable bridge — the long-term architectural memory that the AI's context cannot be. | Opacity Problem, Maintenance Cliff |
| Authority | The standing to explain, decide about, and take responsibility for a system — including who it was built for and why. Requires understanding — ownership without comprehension is commissioning, not authorship. The word does double duty: knowledge sufficient to explain, and standing sufficient to decide. Authority is what Invariant 0 measures. Exists in two dimensions: private authority (can explain to oneself and the AI) and public authority (can explain to the world in publishable prose). The synthesis essay tests the public dimension. | Common Ground, Epistemic Gate, Product Discovery, Public Authority, Synthesis |
| Product Debt | The accumulated gap between what a system does and what users actually need. Analogous to technical debt but invisible in the codebase — it hides in the mismatch between system affordances and user mental models. Product debt compounds as users build workarounds and expectations calcify. | Product Maintenance Cliff, Product Conformance, User Mental Model |
| Product Maintenance Cliff | The point where accumulated product debt makes iteration harder than starting over. Product assumptions encoded in architecture become exponentially more expensive to correct (Boehm's 10-100x cost curve applied to product decisions). Parallel to the technical Maintenance Cliff. | Product Debt, Maintenance Cliff, Inversion Principle |
| Product Discovery | An RDD phase positioned between RESEARCH and MODEL that surfaces user needs, stakeholder maps, value tensions, and assumption inversions. Operates in three modes: forward (greenfield), backward (first-time audit of an existing system with no prior product discovery), and update (refining an existing product discovery artifact against new research — the most common mode in ongoing projects). Its artifact is written in user language and serves as the Rosetta Stone between product vocabulary and system vocabulary. | Phase, Product Vocabulary, Stakeholder Map, Value Tension, Assumption Inversion |
| Stakeholder Map | A section of the product discovery artifact that identifies everyone affected by the system — both direct stakeholders (who interact with it) and indirect stakeholders (who are affected without interacting). Broader than personas, which represent ideal users. From Friedman's Value Sensitive Design. | Product Discovery, Value Tension |
| User Mental Model | A user's internal representation of how the domain works, expressed in their own language. Mapped during product discovery before the domain model maps how the system thinks the domain works. Divergence between the two is a design signal to be surfaced, not a bug to resolve. From Norman. | Product Discovery, Product Vocabulary, Domain Model |
| Value Tension | A conflict between stakeholder needs, or between user needs and business goals, surfaced during product discovery and passed forward as an open question for DECIDE. Value tensions are held, not resolved prematurely. From Haraway's "staying with the trouble." | Product Discovery, Stakeholder Map |
| Assumption Inversion | The output of applying the Inversion Principle: for each major product assumption, the inverted form and its implications. Produced as a section of the product discovery artifact and used to enrich behavior scenarios in DECIDE. | Inversion Principle, Product Discovery |
| Inversion Principle | A cross-cutting epistemological practice: systematically question assumptions before encoding them in architecture. Inverting assumptions is a fundamental research methodology — it surfaces hidden premises by asking "what if this were wrong?" Product assumptions are the most critical application because they hide in the gap between system affordances and user needs, but the principle applies wherever assumptions can silently harden into structure. Operates at four levels: as a procedural step within /rdd-discover, as a habit of mind across RESEARCH, DECIDE, and ARCHITECT, as a narrative framing technique in SYNTHESIS (inverting the obvious takeaway, the process-vs-product assumption, and the reader's assumed context), and as a play technique (the gamemaster introduces inversions to challenge the practitioner's assumptions about their own design). The mechanism serving the product dimension of Invariant 0. | Assumption Inversion, Product Discovery, Authority, Synthesis, Play, Gamemaster |
| Product Vocabulary | Terms in user-facing language that describe the domain as stakeholders experience it. Distinct from system vocabulary (domain model terms). Product vocabulary is captured in the product discovery artifact and traced into the domain model via a provenance column. | Product Discovery, User Mental Model, Domain Model |
| Product Conformance | A higher-level conformance audit that checks the system's product assumptions against actual user needs — complementing technical conformance (which checks code against ADRs). Without product conformance, technical conformance ensures the system is consistently wrong. Distinct from Conformance Audit (which checks artifact template/format alignment). | Product Discovery, Product Debt, Conformance Audit |
| Artifact Legibility | The degree to which an artifact is comprehensible to non-technical stakeholders. Technical artifacts (invariant tables, ADR templates) have low legibility. The product discovery artifact has high legibility because it is written in user language. The orientation document has maximal legibility — it must be comprehensible to *all* audiences (product, technical, and newcomers) because it serves as the universal entry point. The legibility gap is a structural problem the product discovery phase and orientation document address. | Artifact, Product Discovery, Product Vocabulary, Orientation Document |
| Scoping | A team lead's partial pipeline run (RESEARCH → ARCHITECT, no BUILD) used as a leadership thinking tool to build deep understanding and produce handoff artifacts. The team lead can "speak to" the artifacts — explain and defend decisions — because they went through the gates. BUILD is handed off so the team gains their own understanding through implementation. Extended by Scoped Cycle, which adds a lifecycle (scope → cycle → graduate) and applies to subsystems, not just team handoff. | Pipeline, Handoff, Authority, Scoped Cycle |
| Pair-RDD | A collaborative mode where two humans run an RDD cycle together, both responding at epistemic gates. Recovers the pair-programming dynamic lost in agentic coding. The gates become a facilitation structure for human-human dialog, not just human-AI dialog. Not yet formalized in the pipeline. | Epistemic Gate, Epistemic Act |
| Research-Engineer-Writer | A stakeholder who uses RDD for research, understanding, and publication — where the goal is deep comprehension of a problem space, building software that embodies that understanding, and writing about the discoveries the process produced. The pipeline's full arc from RESEARCH through SYNTHESIS serves this stakeholder. The writer dimension processes RDD at a meta-level: each synthesis essay is potentially a contribution to the methodology's own evolution, not just its application. | Phase, Pipeline, Scoping, Synthesis, Public Authority |
| Handoff | The act of passing artifacts to a team or specialists. In the scoping use case, the team lead hands off system design, product discovery, and roadmap artifacts after running through ARCHITECT. Before team handoff, the team lead may share artifacts with architecture and product specialists for external review. The roadmap and field guide make handoff more structured — the team receives sequencing context and implementation mapping, not just specifications. Real workflow, not yet formalized in the pipeline. | Scoping, External Review, Artifact, Roadmap, Field Guide |
| External Review | Specialist feedback (from architecture or product experts) that flows back into the pipeline before team handoff. The team lead shares artifacts for validation, and feedback re-enters at the relevant phase (product feedback at PRODUCT DISCOVERY, technical feedback at ARCHITECT). Confirmed as real workflow but not yet formalized as a pipeline operation. | Handoff, Product Discovery, Pipeline |
| Synthesis | An RDD phase that extracts publishable insight from the artifact trail. The agent mines the trail for novelty signals and conducts a structured conversation (journey review, novelty surfacing, framing) to help the user find the story worth telling. The deliverable is a co-produced outline; the user writes the essay, gives the talk, or builds the exhibition. Not a gate — has a different cost structure (conversation bounded at 30-60 minutes; the communicative act is the user's own work, on their own time). Available after BUILD or after whatever phase terminates the current pipeline run. Usually terminal, but can re-enter RESEARCH when structural experimentation surfaces new questions the pipeline has not addressed. The framing conversation navigates four dimensions: discovery type, narrative form, audience/medium, and epistemic posture — through structural experiments, not purely discursive selection. | Phase, Synthesis Essay, Artifact Trail, Novelty Signal, Dead Discovery, Narrative Form, Epistemic Posture, Structural Experiment, Audience Constraint |
| Synthesis Essay | The user-produced communicative act from the synthesis outline — an essay, talk, presentation, performance, or other realization. A dual-purpose artifact: publishable narrative for the world and narrative context rollup for the pipeline. Produced in the user's own voice — the agent does not produce it. Captures *understanding* (the story of why things are the way they are) where domain models capture vocabulary, system designs capture architecture, and ADRs capture decisions. The final epistemic act of the RDD cycle — and each realization is also a research act, potentially revealing what the writer does not yet understand. | Synthesis, Narrative Context Rollup, Public Authority, Writing as Inquiry, Narrative Form |
| Artifact Trail | The accumulated corpus of artifacts across an RDD cycle — essays, research logs, domain models, ADRs, product discovery documents, system designs, reflections, and code. Technically rich but narratively inert: serves the next pipeline phase but tells no compelling story. The synthesis phase reads across the trail to find what is worth writing about. | Artifact, Synthesis, Dead Discovery |
| Dead Discovery | A genuine insight produced by the RDD pipeline that remains entombed in artifacts — invisible to the user and the world. Dead discoveries are real but buried in the process, distributed across artifacts, and obscured by the technical framing that made them useful within the pipeline. The synthesis phase exists to extract them. | Artifact Trail, Synthesis, Novelty Signal |
| Novelty Signal | A structural marker in the artifact trail indicating genuine surprise or discovery. Five types: (1) explicit surprise statement (language marking violated expectations), (2) reframing event (problem definition changed, not just the answer — Dorst's Abduction-2), (3) assumption denial (finding that denies an assumption the audience holds — Davis's "That's Interesting!"), (4) negative case integration (counter-example that forced revision of the working account), (5) superseded decision (position explicitly revised or replaced). The synthesis agent uses these as a detection sweep to produce a ranked list of discoveries ordered by interestingness. | Dead Discovery, Synthesis, Artifact Trail |
| Volta | The moment in the RDD journey where the writer's understanding genuinely shifted — often the structural center of the synthesis essay, though the writer determines its placement and prominence. Not "the solution was found" (that is resolution); the volta is the discovery that the question itself was wrong, or that the assumed framing concealed the real insight. From poetry (the rhetorical turn in a sonnet). The volta is discovered, not prescribed — the writer's impulse determines where and how it appears in the narrative, or whether to structure around a different principle entirely. The synthesis essay is the writer's work; the volta belongs to the writer's experience. | Synthesis Essay, Novelty Signal |
| Narrative Context Rollup | The practical function the synthesis essay serves within the pipeline: answering "what was this project actually about?" for both human readers and future AI sessions. Where structured artifacts answer "what was decided?" the narrative context rollup answers "what was discovered, and why does it matter?" Addresses the context window ceiling by providing a narrative thread through the artifact corpus. | Synthesis Essay, Context Window Ceiling, Artifact Trail |
| Public Authority | The dimension of authority tested by the synthesis phase: can the user explain the project to the world, in prose that is interesting enough to be worth reading? The epistemic gates throughout the pipeline build private authority (the user can explain to themselves and the AI). The synthesis essay tests whether that understanding can be communicated publicly. A higher bar, and an optional one — but the bar that matters for the research-engineer-writer stakeholder. | Authority, Synthesis Essay, Research-Engineer-Writer |
| Cross-Project Synthesis | A mode of synthesis where multiple RDD cycles across different projects are reviewed simultaneously. The most interesting essays may not be about any single project but about what the projects collectively reveal — concepts that emerged, cross-domain connections that appeared, patterns visible only in the portfolio view. The annotated portfolio mode. | Synthesis, Artifact Trail |
| Writing as Inquiry | The principle (from Richardson) that writing is a method of discovery, not reporting. The synthesis essay is written *toward* a conclusion the writer cannot yet state, not *up* from conclusions already reached. The draft that surprises the writer is the draft that will be interesting to readers. Grounds the synthesis phase's epistemic architecture — the essay is literally the final act of discovery, the moment distributed understanding crystallizes into communicable knowledge. | Synthesis Essay, Epistemic Act |
| Orientation Document | An agent-maintained, living artifact that sits at the top of the artifact hierarchy. Answers "what is this system, who is it for, why does it exist, and where do I go next?" in the fewest words possible. Bridges product-centered and system-centered language without duplicating either — a Rosetta Stone between the two primary readable documents. Derived from the artifact corpus, not authored independently. Its clarity over time serves as a stewardship signal: a document that grows increasingly abstract or internally contradictory indicates the system may be accumulating the wrong kind of complexity. A pragmatic artifact (Invariant 3): the agent generates it, the user validates it. Named ORIENTATION.md — the term emerged naturally throughout research and modeling conversations (ADR-019). | Artifact Hierarchy, Artifact Legibility, Product Discovery, Context Window Ceiling |
| Artifact Hierarchy | The three-tier structure of the RDD artifact corpus. Tier 1: orientation document (entry point — routes readers). Tier 2: primary readable documents (product-discovery.md, system-design.md, roadmap — read end-to-end). Tier 3: supporting material (domain model, essays, ADRs, scenarios, field guide — consulted as needed for provenance and depth). Replaces the prior two-tier model (primary readables + supporting material) by adding the orientation tier above. | Orientation Document, Artifact, Artifact Legibility, Roadmap, Field Guide |
| Roadmap | A strategic document containing work packages with classified dependencies and transition states that enables developer-owned sequencing decisions without dictating build order. Adapted from TOGAF's Architecture Roadmap (Phase E) — "the definition, but not the activities." A pragmatic artifact (Invariant 3): agent-generated, user-validated, updated reflexively. Sits at Tier 2 of the artifact hierarchy, linked from system-design.md. Serves Invariant 0 from the planning side: the developer or team lead can conceptualize how the work gets done. | Work Package, Transition State, Dependency Classification, Artifact Hierarchy, Handoff |
| Field Guide | A Diátaxis explanation-type document that maps the system design's desired state to the current implementation state. Scaffolds mental model construction by connecting domain concepts to code, surfacing design rationale code cannot carry, and pointing to specific files and patterns for exploration. Does not contain understanding — enables the reader to build their own. A pragmatic artifact (Invariant 3): agent-generated, user-validated, updated reflexively. Sits at Tier 3 of the artifact hierarchy as reference material. Serves Invariant 0 from the implementation side: the developer who reads the field guide and explores the code it points to should be able to speak with authority about the system without AI assistance. Pairs with Field Notes during play: the guide is the map of the territory; the notes are the journal of the journey. | Orientation Document, Artifact Hierarchy, Authority, Handoff, Field Notes |
| Work Package | A logical group of changes with dependencies, objectives, and value. The unit of the roadmap. Each work package has classified dependency edges connecting it to other work packages. Borrowed from TOGAF, adapted to RDD's scale. | Roadmap, Dependency Classification, Transition State |
| Transition State | An intermediate architecture between current state and target state that is coherent on its own — not just a step toward the target. Described in the roadmap to show that partial progress produces a working system, not a half-built one. | Roadmap, Work Package |
| Dependency Classification | The scheme for classifying dependency edges between work packages in the roadmap. Three types: **hard dependency** (Module B literally cannot be built without Module A's interface — structural necessity), **implied logic** (Module A is simpler and builds confidence, but Module B could be built first — suggested ordering), **open choice** (Modules C and D are genuinely independent — build either first). Making this classification visible is what distinguishes a roadmap from a build plan. | Roadmap, Work Package |
| Document Sizing Heuristics | Five cascading heuristics governing when documents should be split, how sections should be structured, and what length constraints apply. In priority order: (1) Purpose Test — one purpose, one audience (from Baker's Every Page is Page One); (2) 3-5 Concept Rule — sections should not exceed working memory (from Cowan); (3) ~5,000 Word Guideline — end-to-end documents should aim to stay under ~5,000 words (from LLM lost-in-the-middle research); (4) Read Contract — reference material can be longer than narrative material (from reading comprehension research); (5) Position-Sensitive Placement — critical information at beginning and end, not middle (from LLM recall patterns). | Artifact, Artifact Hierarchy |
| Conformance Audit | A systematic scan comparing the current RDD skill version's expected artifact structure against the project's actual artifact corpus. Produces a gap analysis — missing artifacts, missing sections, template mismatches, broken inter-artifact references — with prioritized remediation. Distinct from Product Conformance (which audits product assumptions against user needs). Four operations: audit, remediation, drift detection, graduation. A pragmatic action (Invariant 3). | Documentation Drift, Graduation, Product Conformance |
| Documentation Drift | The divergence between what RDD's skill files expect and what the project's artifact corpus actually contains. Analogous to architectural drift (code diverging from architecture) but applied to documentation. Three causes: skill evolution (new artifact types, amended templates), RDD adoption (adding RDD to an existing project), and post-build drift (code changes without corresponding doc updates). The conformance audit's target problem. | Conformance Audit, Artifact |
| Graduation | Rolling up RDD-produced knowledge (architectural decisions, domain vocabulary, product context, key rationale) into the project's native documentation format and archiving the RDD-specific artifacts as historical record. A recurring operation at the subsystem level — not a one-time exit from RDD. Applies Invariant 6 (scaffolding must fade) at the methodology level: once the understanding is built and integrated, the RDD scaffolding comes down until the next problem warrants it. Distinct from abandoning RDD — the archived artifacts remain available if a future cycle is needed. | Scoped Cycle, Documentation Fatigue, Scaffolding, Fading, Conformance Audit, Deep Work Tool |
| Scoped Cycle | Running an RDD cycle on a subsystem or feature rather than the whole project. The natural pattern once project-level architecture stabilizes: the user scopes RDD to a subfolder in docs, runs the cycles needed, then graduates the knowledge into project-level docs. Extends Scoping (partial pipeline for team handoff) with a lifecycle: scope → cycle → graduate. The subsystem's domain model and system design respect the parent project's constraints without duplicating the entire project context. | Scoping, Graduation, Pipeline, Deep Work Tool |
| Documentation Fatigue | The felt experience when RDD artifacts outlive their immediate value and become maintenance burden. A design signal — not a failure of the methodology, but an indication that the work has moved past the point where RDD's structure adds value at that scope. The appropriate response is graduation and re-scoping, not abandoning discipline. | Graduation, Scoped Cycle |
| Deep Work Tool | The framing of RDD's role within a project's broader workflow: a tool for structured thinking on complex problems that integrates into existing practices rather than replacing them. RDD is something you reach for when a problem warrants it and put away when the knowledge has been absorbed. The project's native documentation is the permanent home; RDD docs are the working space. This framing makes graduation natural (folding knowledge back into native docs) and scoped cycles the default (applying RDD where it's most needed). | Graduation, Scoped Cycle, Pipeline |
| Narrative Form | The structural shape a telling takes. Ranges from established structural patterns (braided, segmented, whorl of reflection, lyric, narrative with a lift, dipping into the well, formal limits of focus) through hermit-crab forms (essay inhabiting a non-essay shell) to meta-forms (Borgesian, speculative nonfiction, counterfactual). Not chosen from a menu; emerges from the framing conversation as the writer discovers what the material demands through structural experiments. One of four dimensions the framing conversation navigates. | Synthesis, Hermit Crab Form, Structural Experiment, Novelty Signal |
| Hermit Crab Form | A narrative form that inhabits the shell of a non-essay genre — algorithm, recipe, field guide, proof, dialogue, literate program, exhibition catalog, autopsy report, audio guide. From Miller & Paola (*Tell It Slant*, 2003). The borrowed form is not decoration; it *generates* content the writer would not have found otherwise by bypassing what the intellectual mind has already determined as "the story." Applies equally to written and performed outputs. A talk can inhabit a borrowed shell too. | Narrative Form, Form-Content Generation |
| Epistemic Posture | The writer's stance toward certainty during synthesis. Three positions: *determined* (know the argument, need the best form to deliver it — outline as map), *exploratory* (sense something, need writing to reveal it — outline as invitation, after Montaigne's *essayer*), *indeterminate* (have elements and relationships, each realization discovers something new — outline as score, after Cage's indeterminacy). The posture determines what the outline even *is*. One of four dimensions the framing conversation navigates. | Synthesis, Volta, Writing as Inquiry |
| Structural Experiment | A quick, externalized sketch in a candidate form — drafting an opening as pseudocode, rewriting the pitch for a five-year-old, arranging artifacts as exhibition stops — done to see what the form reveals about the material. Not a commitment. Failures are diagnostic: the mismatch between form and material clarifies what the material actually needs. The mechanism by which the framing conversation avoids being purely discursive. Grounded in ensemble-based devising traditions (Viewpoints, Laban): discovery comes from the outside, from externalized structural experimentation, not from thinking about structure. | Frame Narrative, Narrative Form, Form-Content Generation |
| Curatorial Arrangement | The experiential structure of the outline, layered over the argumentative backbone. Meaning through selection, juxtaposition, scale shifts, negative space, and the personal voice of the curator-writer. An exhibition that "works" is not necessarily one where every claim follows deductively — it is one where the juxtapositions create a coherent emotional and intellectual experience. From the wunderkammer tradition and the Museum of Jurassic Technology (Weschler's "shimmer" — the productive state between wonder and doubt). | Argumentative Backbone, Synthesis, Artifact Trail |
| Argumentative Backbone | The logical structure of the outline that the argument audit verifies — claims follow from evidence, framing does not overreach. Exists even in lyric or hermit-crab forms; the logic may be subterranean but must be sound. Distinct from the curatorial arrangement, which creates the experiential structure. The argument makes it true; the curation makes it alive. | Curatorial Arrangement, Synthesis |
| Form-Content Generation | The principle that formal constraint generates content — it does not merely contain pre-existing content. The form is an instrument of discovery. Stated as a literary technique by OuLiPo and Miller & Paola (hermit crab), but the stronger claim comes from ensemble-based devising traditions: discovery comes from the outside, from externalized structural experimentation, not from thinking about structure. The core mechanism behind structural experiments, hermit crab forms, and audience constraints. | Hermit Crab Form, Structural Experiment, Audience Constraint |
| Audience Constraint | Audience and medium treated as a creative catalyst rather than a delivery specification. Each medium imposes constraints (Story Collider: 10 min, no slides, personal story; Ignite: 5 min, 20 slides auto-advancing; explain-to-a-child: simple words only) that function like formal constraints — forcing creative compression and surfacing content that unconstrained composition misses. The audience may not be known at the start; the material may disclose its audience through the act of composition. One of four dimensions the framing conversation navigates. | Synthesis, Form-Content Generation, Structural Experiment |
| Plugin | A self-contained distribution unit packaging skills, agents, hooks, and artifacts for installation via Claude Code's plugin system. RDD as a plugin ships under the name `rdd` in the `nrgforge` organization. Namespaces skills as `/rdd-research`, `/rdd-decide`, etc. The plugin directory is copied to the user's local cache on installation; the full artifact corpus ships with it. | "Plugin" / "install the plugin" | Specialist Subagent, Hook, Artifact |
| Specialist Subagent | An AI agent running in an isolated context window that reads input artifacts, performs focused analysis or generation, and writes output artifacts. Has no conversation history and no user interaction. May run on a different model than the orchestrator. Six proposed: citation-auditor, argument-auditor, lit-reviewer, conformance-scanner, orientation-writer, spike-runner. | "Dispatch" | Orchestrating Agent, Audit Artifact, Artifact-Mediated Communication |
| Orchestrating Agent | The main-conversation AI agent that runs phase skills, manages epistemic gates, dispatches specialist subagents, and interacts with the user. During the play phase, takes on the gamemaster role — active facilitation rather than skill execution or artifact production. The existing AI Agent role, named to distinguish from specialist subagents. | — | Specialist Subagent, Epistemic Gate, Gamemaster |
| Hook | A passive event handler configured in JSON that fires on system events (PreToolUse, PostToolUse, UserPromptSubmit, Stop, SubagentStart, SubagentStop, etc.). Enforces cross-cutting concerns without embedding logic in every skill. Five proposed for the RDD plugin: invariant reminder, epistemic gate enforcer, skill activator, orientation trigger, document sizing check. | "Skill activator" | Skill Activator, Invariant, Epistemic Gate |
| Audit Artifact | A durable, inspectable file produced by an auditor subagent (citation-auditor, argument-auditor, conformance-scanner). Structured as summary counts followed by prioritized issues (P1 must-fix, P2 should-fix, P3 consider) with locations, evidence, and recommendations. Follows the artifact-mediated communication pattern — exists as a file in the artifact trail, not as ephemeral conversation output. | "Audit report" / "audit artifact" | Specialist Subagent, Artifact Trail, Artifact-Mediated Communication |
| Artifact-Mediated Communication | The design principle that specialist work communicates through artifact files rather than conversation context. Extends RDD's existing inter-phase pattern (phases communicate via essays, domain models, ADRs) to intra-phase specialist work. The motivation is architectural consistency and artifact durability, not context window savings — with 1M token windows, context is not scarce, but ephemeral output is still uninspectable and unreferenceable. | — | Specialist Subagent, Audit Artifact, Artifact |
| Skill Activator | A UserPromptSubmit hook that detects RDD-related prompts and suggests the appropriate phase skill. | Play | A post-BUILD phase where the practitioner inhabits stakeholder roles and freely explores the built system, discovering what specifications missed through situated encounter. Three movements: inhabit → explore → reflect. Structurally parallel to synthesis — play itself is the epistemic act (ADR-016 principle extends), bounded by felt understanding (Invariant 0) rather than timebox (Invariant 4). Distinct from testing at the ontological level: testing is practitioner-directed verification; play is disclosure by encounter (Gadamer). The discover-play symmetry gives the pipeline epistemic bookends around the pragmatic center — discover inverts assumptions about what to build, play inverts assumptions about what was built. Distinct from exploratory testing: exploratory testing asks "where might this fail?" within the verification frame; play attends to whatever the system discloses in Bateson's play frame. | Phase, Field Notes, Gamemaster, Interaction Specification, Product Discovery, Synthesis, Epistemic Distance, Playable Surface, Point of Concentration |
| Interaction Specification | The missing middle layer between business-rule scenarios and technical implementation, specifying how each stakeholder works with the system at the workflow level. Fills the persona-to-actor gap: derives task decompositions from product discovery's stakeholder models. Corresponds to Adzic's workflow layer, the Screenplay Pattern's Task layer, and HTA's operation level. The derivation method from stakeholder model to task decomposition is an open design problem — the contribution at this stage is identification of the gap and the architectural precondition (product discovery provides the upstream input that no existing methodology supplies), not the complete pipeline. Creates the playable surface that the play phase requires. | Scenario, Play, Product Discovery, Playable Surface, Super-Objective, Objective |
| Gamemaster | The agent's role during the play phase: active facilitation that proposes stakeholder roles and points of concentration, introduces complications and inversions, reacts to player discoveries, and creates conditions for genuine encounter without directing the outcome. Named by analogy to tabletop RPG facilitation and Spolin's side-coaching. Sits on the Invariant 3 boundary: pragmatic in form (the agent performs it), epistemic in function (shapes what the practitioner attends to and encounters). Distinct from the orchestrating agent's other modes — not executing skills (procedural), not producing artifacts (pragmatic), but actively facilitating during the practitioner's epistemic act. A human gamemaster (colleague, UX researcher, domain expert) is equally valid for RDD the method; RDD the plugin implements the agent in this role. | Play, Orchestrating Agent, Point of Concentration, Epistemic Distance, Side-Coach |
| Field Notes | Play's artifact: observational, naturalistic, in-the-moment discovery records categorized by feedback destination (missing scenarios → DECIDE, usability friction → DISCOVER, new questions → RESEARCH, challenged assumptions → DISCOVER, delight → SYNTHESIS, interaction gaps → interaction specs). Not a test report — play observes and records but does not prescribe fixes. Fixes go through the normal cycle (Bruner's reduction in the seriousness of consequences). Pairs with the Field Guide: the guide is the map of the territory; the notes are the journal of the journey. | Play, Field Guide, Artifact |
| Point of Concentration | A single focusing constraint that paradoxically enables freedom within play. From Spolin's improvisation pedagogy. Each play session has a focus — a stakeholder, a job, an inversion — but freedom within that focus. Resolves the paidia-ludus tension (Caillois): enough structure to be productive, enough freedom to discover. Analogous to how synthesis uses structural experiments: the framing conversation provides focus but the experiments are free within those dimensions. | Play, Gamemaster |
| Epistemic Distance | The structural tension in the play phase: can the person who built the system genuinely inhabit unfamiliar stakeholder roles and encounter their own design with fresh eyes? From Deterding's critique of Sicart. Mitigated by three mechanisms: the play frame itself makes the familiar strange (Bateson's metacommunicative shift), product discovery's stakeholder profiles provide concrete anchoring material for inhabitation, and the gamemaster introduces complications the builder would not self-generate. Reduces the problem from "imagine a user you've never met" to "inhabit a stakeholder whose needs and mental model discover has already surfaced." Playing as oneself (when the practitioner is the primary stakeholder) eliminates the distance concern entirely — the play frame does the distancing regardless of whose role is inhabited. | Play, Gamemaster, Product Discovery |
| Playable Surface | What interaction specifications create: a structured layer of specified interaction mechanics that gives play something concrete to encounter and discover deviations from. Without a playable surface, play is frustrating because there is nothing structured against which to discover divergence. Confirmed through real user frustration (nrgforge/rdd#7). The ordering holds: discover → interaction specs → build → play. | Interaction Specification, Play |
| Super-Objective | A stakeholder's overarching need, drawn from product discovery's stakeholder map and jobs. The top level of play's inhabitation structure (Stanislavski). Provides the motivational anchor for the practitioner when inhabiting a role: not just "be this person" but "pursue what this person fundamentally needs." | Play, Product Discovery, Objective |
| Objective | What a stakeholder wants in a given interaction, drawn from the interaction specification layer. The middle level of play's inhabitation structure (Stanislavski). More specific than the super-objective, more situated than a scenario. | Play, Interaction Specification, Super-Objective, Obstacle |
| Obstacle | What gets in the way of a stakeholder's objective. Discovered through play, not specified in advance. The third element of play's inhabitation structure (Stanislavski). Obstacles are the raw material of play's discoveries — friction, confusion, competing affordances, unstated assumptions. | Play, Objective, Field Notes |
| Reflection Time | The user-facing name for an Epistemic Gate. Signals pause-and-think rather than test-and-pass. The agent introduces it: "Before moving on — reflection time." Followed by an adaptive prompt calibrated through the Attend-Interpret-Decide cycle. "Epistemic gate" remains in research and design vocabulary; "reflection time" faces the user. Grounded in Schon's reflection-on-action and Rowe's wait-time research. | "Reflection" / "reflection point" / "pause" | Epistemic Gate, Attend-Interpret-Decide, Reflection |
| Attend-Interpret-Decide | The three-phase cycle the agent runs at each gate, replacing fixed-template prompt selection. Attend: read the cycle's conversation history for engagement signals. Interpret: form a hypothesis about the user's engagement and understanding. Decide: select a pedagogical move calibrated to the interpretation. Grounded in the teacher noticing framework (Jacobs, Lamb & Philipp, 2010). | "AID cycle" / "adaptive gate" | Epistemic Gate, Engagement Signal, Pedagogical Move, Contingent Teaching |
| Engagement Signal | An observable marker in the conversation history indicating the quality of the user's engagement during a phase. Deep engagement markers: authentic questions, elaborated explanations with reasoning connectives, specificity referencing artifact content, challenges and alternatives, connections to prior knowledge. Surface compliance markers: brief responses without elaboration, approval without specificity, restating without adding, not building on prior discussion. Quality, not quantity, predicts learning outcomes. | — | Attend-Interpret-Decide, Pedagogical Move |
| Pedagogical Move | The agent's contingent response at a gate, selected by the Attend-Interpret-Decide cycle. Five modes: **Challenge** (deep engagement — surface tensions, apply inversions, reframe), **Probe** (adequate engagement — ask for reasoning and specificity), **Teach** (thin engagement — explain the key choice, then ask), **Clarify** (confusion — back up, address misunderstanding), **Re-anchor** (disengagement — connect to what the user is building, or suggest a break). Challenge is the response to deep engagement, not to poor engagement. | — | Attend-Interpret-Decide, Contingent Teaching, Contingent Shift |
| Contingent Teaching | The four-step iterative loop the agent runs within a gate when teaching is needed: (1) diagnose the user's understanding, (2) check the diagnosis with the user, (3) intervene with targeted explanation, (4) check whether understanding moved forward. Not a one-shot exchange — the loop repeats until shared understanding is established. From the scaffolding literature (Wood, Bruner & Ross, 1976). The agent-as-teacher is not a separate mode; it IS the response when the Interpret step reveals a gap. | "Teaching mode" | Pedagogical Move, Contingent Shift, Epistemic Gate |
| Contingent Shift | The mechanism that makes gates adaptive: increase support when the user struggles, decrease support when they succeed. From Wood, Bruner & Ross (1976). Creates a "region of sensitivity" where the user never succeeds too easily or fails too often. Applied at gates: thin response → shift toward teaching; rich response → shift toward challenge. Applied across cycles via Fading (Invariant 6): early cycles involve more teaching, later cycles more challenge. | — | Contingent Teaching, Pedagogical Move, Fading |
| Earned Fatigue | Tiredness from genuine deep engagement with the domain during an RDD cycle. A signal to take a break, not a signal that the methodology is too heavy. Qualitatively different from the scattered exhaustion of context-switching across multiple AI agents. When the agent detects fatigue (thin responses following earlier deep engagement), the appropriate response is to name it gently and offer a break or reframe. | "Earned fatigue" / "good tired" | Engagement Signal, Pace Regulator |
| Pace Regulator | RDD's function as a counter-practice to the AI productivity loop — constant agent spinning, context switching while waiting on output, the dopamine of "almost done." Gates anchor the user in one domain, one problem, one thread of understanding. The value proposition is not productivity but the felt experience of understanding. RDD cannot be sold on productivity grounds; it must be experienced as a different kind of work. | "Deep work" / "staying focused" | Deep Work Tool, Earned Fatigue, Epistemic Gate |
| IRE Pattern | The Initiation-Response-Evaluation discourse structure (Sinclair & Coulthard, 1975; Mehan, 1979): teacher asks a display question, student responds, teacher evaluates. The dominant but most criticized classroom discourse pattern. RDD's current gates replicate this structure: the agent asks a predetermined prompt, the user responds, the agent notes discrepancies and proceeds. The Attend-Interpret-Decide cycle replaces this with referential questions (genuinely open) and elaborative feedback (building on the response). Avoid: this is the pattern being replaced. | — | Approval Gate, Epistemic Gate, Attend-Interpret-Decide |
| Reframing | Questioning whether the problem as stated is the right problem, stepping outside the solution space to consider alternatives. The agent can make this move at gates because it holds cross-phase context the user may have lost across sessions. "The research explored approaches A, B, and C, but the domain model suggests the core problem is about X." A higher-order application of the Inversion Principle. | "Stepping back" / "zooming out" / "have we considered..." | Inversion Principle, Pedagogical Move, Attend-Interpret-Decide |
| Code Review | An epistemic activity where the reviewer builds genuine understanding of code changes — not primarily defect-finding but knowledge transfer, shared understanding, and reviewer learning. Research (Bacchelli & Bird, ICSE 2013) shows defect-related comments comprise ~12.5% of review discussion; the primary outcomes are code improvement, knowledge transfer, and team awareness. Within RDD, operates as a utility invocable during build stewardship or standalone. Two modes: corpus-grounded (within an RDD artifact trail) and context-reconstructive (standalone, RDD-lite). Two cognitive phases: orientation (~27%) and analysis (~73%), per the CRDM model. Success criterion: the reviewer can discuss the changes with informed judgment without AI assistance afterward. Note: this is Invariant 0 adapted to the review context — the reviewer is not the author and should not need to be. The bar is "speak with informed judgment" (enough understanding to converse genuinely about the changes), not "speak with authority" (the builder's bar). The reviewer brings human-centered evaluation: is this the right thing to build in the right way given our goals? | "Help me be helpful" | Review Orientation, Review Analysis, Corpus-Grounded Review, Context-Reconstructive Review, Review Question, Reviewer-as-Passthrough, Opacity Problem |
| Review Orientation | The first cognitive phase of code review (~27% of verbalized cognitive effort per the CRDM model), where the reviewer establishes context: why the change exists, what it intends to accomplish, what constraints shaped the approach. Most review tools skip this phase, sending the reviewer straight to a diff. In corpus-grounded mode, RDD artifacts scaffold orientation; in context-reconstructive mode, the review process builds orientation from tickets, docs, and discussion through collaborative context-gathering. | "Orientation" | Review Analysis, Corpus-Grounded Review, Context-Reconstructive Review, Collaborative Context-Gathering |
| Review Analysis | The second cognitive phase of code review (~73% of verbalized cognitive effort per the CRDM model), where the reviewer iterates through cycles of understanding, assessing, and planning. Depends on pattern recognition built from experience — effective analysis requires adequate orientation. Maps to Klein's Recognition-Primed Decision model (the same framework describing decision-making by fireground and military commanders). | — | Review Orientation, Code Review |
| Review Question | The primary output of the review skill: questions that guide the reviewer's thinking rather than findings that tell them what to think. "Based on this context, what questions should I be asking?" Structurally different from findings — findings produce verdicts, questions produce understanding. Mechanical findings (genuine bugs, missing null checks) are surfaced separately as a distinct, secondary output type. | "Questions, not findings" | Code Review, Scaffolding |
| Corpus-Grounded Review | Review mode operating within an RDD artifact trail, scoped to a work package. Research essays answer "why does this problem matter?", ADRs record decisions and rationale, scenarios provide refutable behavior specs, the domain model provides ubiquitous language. The reviewer inherits a provenance chain that grounds evaluation in context rather than guesswork. | "Corpus-grounded" | Review Orientation, Work Package, Artifact Trail, Code Review |
| Context-Reconstructive Review | Review mode operating outside an RDD corpus — reviewing a colleague's MR, evaluating unfamiliar code. The review process performs lightweight discovery: the reviewer provides breadcrumbs (ticket URLs, MR links, doc pointers) and the agent follows them (fetching via CLI/MCP tools, reading, synthesizing). A mini-discovery that reconstructs the orientation context the CRDM model shows is essential. User-facing name: RDD-lite. | "Context-reconstructive" / "RDD-lite" | Review Orientation, Product Discovery, Collaborative Context-Gathering, Code Review |
| Collaborative Context-Gathering | The opening step of code review where the reviewer provides pointers (ticket URLs, doc links, discussion threads) and the agent fetches and synthesizes them. Neither fully automated nor fully manual — the reviewer knows *what* to fetch, the agent *can* fetch. Within RDD, the work package scopes this naturally. Outside RDD, the reviewer's domain knowledge determines the breadcrumbs. | — | Context-Reconstructive Review, Review Orientation |
| Reviewer-as-Passthrough | Anti-pattern: receiving AI-generated review comments, forwarding them without evaluation, and claiming the review as one's own work. Achieves perfect review metrics while producing zero understanding. A team-level manifestation of the Metacognitive Illusion — the reviewer believes review happened because comments were generated. The review happened on paper; it did not happen in anyone's mind. | "Reviewer-as-passthrough" | Metacognitive Illusion, Code Review, Reverse Centaur |
| Reverse Centaur | Anti-pattern where the human serves the AI pipeline rather than the AI serving the human. In the proper centaur model, the human provides strategic judgment and the AI handles tactical computation. In the reverse centaur, the human rubber-stamps AI output at the speed the pipeline demands — nominally in the loop, actually serving the machine's workflow. Compounded by the accountability asymmetry: the AI generates, but the human is liable. | "Reverse centaur" | Reviewer-as-Passthrough, Accountability Asymmetry, Opacity Problem |
| Accountability Asymmetry | The structural property that AI generates code but the human is liable for what ships. The reviewer's name is on the approval. Understanding what was approved is not just epistemically virtuous but professionally protective — the reviewer who genuinely understood can explain and defend their decision; the one who passed through AI comments cannot. | "Accountability asymmetry" | Reverse Centaur, Code Review, Authority |
| Deskilling Spiral | The risk that over-reliance on AI-assisted review erodes the human judgment it ostensibly supports. Reviewers who delegate judgment to AI lose the ability to exercise it. Supported by analogy to medical literature on AI-induced deskilling (suggestive, not yet proven for software). Genuine code review preserves capability by exercising it. | — | Reverse Centaur, Reviewer-as-Passthrough, Fading |
| Guardrails Paradox | The tension that automated safety nets (CI/CD, test coverage, static analysis) designed to compensate for reduced human attention also signal that human attention is less necessary. The safety net becomes the excuse not to look down. The reviewer is squeezed: expected to ship fast against rising AI-generated code volume AND expected to be the last line of defense when guardrails miss something. | "The guardrails paradox" | Code Review, Reverse Centaur, Opacity Problem |
| Zone of Proximal Development | The space between what the reviewer can understand alone and what they cannot understand at all — where the review skill operates. From Vygotsky. The skill does not make the reviewer an expert; it helps them reach *enough* genuine understanding to be useful, in the time available. The ZPD shifts with available time: deep orientation when there is an hour, the single most important question when there are five minutes. Variable depth, consistent quality. | "Zone of proximal development" | Scaffolding, Fading, Code Review, Review Orientation |
| Productive Resistance | This project's interpretive synthesis (Essay 011): the friction that forces a developer to externalize and revise their mental model. Draws from five traditions (Papert's constructionism, Schön's reflection-in-action, Ericsson's deliberate practice, Kirsh & Maglio's epistemic actions, debugging research) but the unifying label is original. Operates in two modes: foundational (orientation gives enough context to start) and distributed (iterative TDD, stewardship, refactoring deepens understanding). Neither mode alone is sufficient. | "Staying sharp," "guardrails" | Epistemic Action, Scaffolding, Opacity Problem |
| Context-Reconstructive Mode | An operating mode for skills (build, debug, refactor, review) where orientation is synthesized from heterogeneous sources (tickets, codebase, discussion threads, docs, verbal description) rather than read from an RDD artifact corpus. The mode that serves most of the world's codebases. Produces Reconstructed Facsimiles of what the pipeline's upstream phases normally provide. | "Context-reconstructive" / "RDD-lite" | Pipeline Mode, Orientation Questions, Context Gathering, Reconstructed Facsimile |
| Pipeline Mode | An operating mode for skills where the full RDD artifact corpus is available (domain model, system design, scenarios, ADRs). The skill benefits from the depth and provenance the corpus carries. Named to contrast with Context-Reconstructive Mode. | "Pipeline mode" | Context-Reconstructive Mode, Artifact Trail |
| Orientation Questions | Five questions that compress the pipeline's essential knowledge into actionable context: (1) Who is this for and why? (2) What are we building? (3) What's the scope, integration points, and interaction mechanics — and how are they testable? (4) What's ambiguous, and does it need stakeholder input or just a decision? (5) How do we demo what was built? A lossy compression — preserves essential orientation at the cost of depth and provenance. | "Orient first" | Context-Reconstructive Mode, Context Gathering, Product Discovery, Roadmap |
| Composable Skill Family | The architectural pattern of four skills (build, debug, refactor, review) that share a Context Gathering foundation and can be used standalone or composed within a build cycle. Each skill represents a distinct epistemic activity. Build composes the others: debug when a bug is encountered, refactor when smells are identified, review at stewardship checkpoints. | — | Context Gathering, Code Review, Three-Level Refactor |
| Context Gathering | The shared protocol used by all skills in the Composable Skill Family before doing their specialized work. | Reconstructed Facsimile | A lightweight artifact produced during Context Gathering in context-reconstructive mode — enough to orient and operate, not a full pipeline artifact. | Three-Level Refactor | The refactor step reconceived for AI-assisted development, operating at three levels: (1) Smells — identify what is wrong using classical (Fowler) and AI-specific catalogs; (2) Patterns — apply the appropriate technique from the refactoring catalog; (3) Methodology — check against architectural intent (module boundaries, dependency rules, domain vocabulary). | Everyday Developer | A stakeholder who uses build, debug, refactor, and review skills outside the full RDD pipeline for everyday work. | Comprehension Debt | The growing gap between how much code exists in a system and how much any human genuinely understands. Term from Osmani (2026). AI generates at 5-7x the rate developers absorb. The Opacity Problem quantified. | — | Opacity Problem, Maintenance Cliff, Intent Debt |
| AI Smell Taxonomy | The three-tier classification of code patterns in AI-generated codebases (Essay 011, original to this work): **Novel** (mechanistically require AI — Constraint Decay, Slopsquatting, Intent Debt), **Exacerbated** (existed in human codebases, AI makes worse via specific mechanism), **Accelerated** (same old problems at higher volume). | Constraint Decay | A genuinely novel AI pattern (name original to this work; underlying interaction smells from arxiv 2603.09701): earlier instructions lose influence as conversation length grows due to context-window positional attenuation. | Slopsquatting | A genuinely novel AI pattern (term from FOSSA/security community): LLMs generate package names from distributional priors, not live registries. ~20% of generated code samples contain non-existent packages. | Bugs Déjà-Vu | An AI-exacerbated pattern (term from Ox Security): identical bugs recur via two mechanisms — session amnesia (no persistent learning) and training-data memorization (44-83% identical bug rates per arxiv 2503.11082). 70-80% prevalence. | — | AI Smell Taxonomy |
| Oracle Mirroring | An AI-exacerbated pattern (name original to this work): tests validate the implementation, not the intent. ## Invariants

0. **The user should be able to speak with authority about what was built, who it was built for, and why.** After completing an RDD cycle, the user should be able to explain the system's design rationale, key decisions, domain vocabulary, and the user needs those decisions serve — without AI assistance. This is the outcome all other invariants serve. If the methodology satisfies invariants 1-7 but the user cannot do this, the methodology has failed.

1. **Understanding requires generation, not review.** Passive review of AI output does not produce durable understanding. Every gate must require the user to produce something — an explanation, prediction, reconstruction, or articulation — not merely assess something.

2. **Epistemic acts are mandatory at every gate.** No phase transition may consist solely of approval. At minimum, one epistemic act must be performed before proceeding.

3. **Pragmatic actions may be automated; epistemic actions may not.** AI handles generation, search, formatting, and first drafts (pragmatic). The user handles explanation, prediction, retrieval, and articulation (epistemic). The boundary between these categories is the core design decision.

4. **Epistemic cost must be productive, not merely brief.** Each gate's time must build understanding. The constraint is on *waste* (formulaic exchanges that build nothing), not on *duration* (a teaching conversation that resolves a real comprehension gap). Productive gates may take longer when the Attend-Interpret-Decide cycle reveals the user needs teaching; this is the methodology working, not failing. If the cost becomes burdensome without being productive, users will circumvent it. Earned fatigue from deep engagement is a signal to take a break, not a signal that the methodology is too heavy.

5. **Approval is not grounding.** Verifying that output looks correct does not establish shared understanding. Epistemic acts function as grounding moves that verify alignment between user intent and AI interpretation.

6. **Scaffolding must fade.** Constant AI assistance at a fixed level is a crutch, not scaffolding. Across RDD cycles, the methodology should shift generative burden toward the user as expertise grows.

7. **Epistemic acts are bidirectional.** Each act simultaneously builds user understanding and enriches AI context for subsequent phases. This is not a side effect — it is a core mechanism. The compounding quality effect depends on both directions operating.
